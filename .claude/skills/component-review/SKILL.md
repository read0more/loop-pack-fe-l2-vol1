# Component Review — 관심사 분리 & Custom Hook 리뷰

컴포넌트가 한 가지 일에 집중하는지, 레이어(UI / 비즈니스 로직 / API)가 분리됐는지를 리뷰한다.

---

## 리뷰 실행 순서

1. **대상 읽기** — 인자로 받은 파일(없으면 사용자에게 대상 파일/컴포넌트를 묻는다)과
   그 컴포넌트가 의존하는 연관 파일(같은 폴더/import된 hook · service · util · types)을 함께 읽는다.
   레이어가 이미 나뉘어 있다면 각 레이어 파일까지 같이 봐야 "분리됐는지"를 판단할 수 있다.
2. **레이어 분류** — 먼저 *관심사 분류 기준*의 레이어 표로 코드를 UI / 비즈니스 로직(Hook) /
   API(Service) / Utils 로 나눈다. 라인 위치를 같이 기록한다.
3. **상태 분류** — 그다음 상태를 다루는 비즈니스 로직(Hook) 안에서 상태가 서버 상태 /
   클라이언트 상태 / 파생값·로직 조합 으로 잘 나뉘었는지 본다.
4. **체크리스트 적용** — 아래 *리뷰 체크리스트* 항목별로 ✅(양호) / ⚠️(개선 후보)를 판정한다.
5. **리포트 출력** — *리포트 출력 형식* 그대로 결과를 낸다. 파일은 수정하지 않고 후보만 제안한다.

---

## 관심사 분류 기준 — 3개 레이어 & 상태 3분할

**① 레이어 분리** — 아래 표 기준으로 레이어를 나눈다. 각 레이어는 **독립적으로 이해 가능**해야 한다.

| 레이어 | 책임 | 질문 |
| --- | --- | --- |
| Components (UI) | JSX 렌더링, 이벤트 바인딩, 스타일, 작은 파생값 | "어떻게 보이는가" — 이 코드만 봐도 화면이 그려지는가 |
| Hooks (비즈니스 로직) | 상태 관리, 필터/정렬/검색, 서버 상태 조합, 유효성 검증 | "어떻게 동작하는가" — 이 코드만 봐도 기능이 이해되는가 |
| Services / Repository (API) | API 호출, 요청/응답 변환, 공통 에러 변환 | "어디서 가져오는가" — 이 코드만 봐도 서버 스펙이 파악되는가 |
| Utils | 상태·Side Effect 없는 순수 함수 | 입력을 출력으로 바꾸기만 하는가 |

**② 상태 3분할** — 레이어를 나눈 뒤, 상태를 다루는 비즈니스 로직(Hook) 안에서 상태가 세 종류로
잘 나뉘었는지 본다. "Custom Hook이 뭐고 상태관리가 뭐냐"는 혼란은 대부분 세 종류를 안 나눠서 생긴다.

| 분류 | 도구 | 예시 | 리뷰 포인트 |
| --- | --- | --- | --- |
| 서버 상태 | **TanStack Query (표준)** — 손수 useEffect+state 직접 구성은 지양/전환 후보 | 주문 내역, 목록, 인증 정보 | 캐싱·재요청·중복 제거·stale 관리가 필요한 비동기 데이터 |
| 클라이언트 상태 | useState · useReducer · Zustand · Jotai | 모달 열림, 선택된 탭, 입력값 | UI 로컬 상태 |
| 파생값 / 로직 조합 | 일반 함수 · Custom Hook(조합) · 렌더링 중 계산 | 필터된 목록, 정렬된 목록, 할인율, 포맷팅 | **state로 만들지 말 것** — 계산하면 되는 값 |

레이어 간 관계: Hook이 서버 상태 · 클라이언트 상태 · 파생값을 **조합**하고, UI는 그 조합된 결과를 받아 렌더링한다.

**③ 두 번째 축 — 컴포넌트 경계·계약·합성** — 레이어/상태를 나눈 뒤, 컴포넌트 자체의 모양을 본다.
세 질문으로 압축된다.

- **경계** — "이 컴포넌트는 **몇 가지 이유로 바뀌나**, 직접 **구현**인가 자식들의 **조합**인가." 하나를 넘으면 자를 자리다.
- **계약(props)** — "props는 **동작**을 주입하는가(구조는 자리로 내준다), 개수는 적고 이름은 **역할대로**인가."
- **합성** — "props를 늘리거나 Context를 쓰기 전에 **children/slot 합성**으로 풀 수 있나."

이 축은 체크리스트 7·8·9 로 이어진다.

## 리뷰 체크리스트

### 1. 레이어 분리
- [ ] 한 컴포넌트에 fetch + 필터 + 정렬 + 페이지네이션 + 에러/로딩 + UI가 한꺼번에 들어있지 않은가?
- [ ] UI 코드만 봐도 화면이 그려지는가? (도메인 계산·API 호출이 JSX 사이에 섞여 있지 않은가)
- [ ] API 호출이 컴포넌트 안에 직접 있지 않고 service / Custom Hook 뒤로 숨겨져 있는가?

### 2. 상태 분류 — 파생값을 state로 만들지 말 것 (최중요)
- [ ] 필터된 목록 · 정렬된 목록 · 포맷 문자열 · 할인율처럼 **계산 가능한 값**을
      `useState` + `useEffect`로 동기화하고 있지 않은가?
- [ ] 판단 질문: **"이 값은 저장해야 하는가, 계산하면 되는가?"** — 계산 가능하면 파생값으로.
- [ ] 서버 상태(원격·비동기)와 클라이언트 상태(로컬 UI)가 한 덩어리로 섞여 관리되고 있지 않은가?

### 3. useEffect — 외부 시스템 동기화 용도인가
- [ ] Effect가 **React 바깥 외부 시스템 동기화**(서버 API, 브라우저 구독, WebSocket, 타이머,
      외부 라이브러리)에만 쓰이는가?
- [ ] props/state로부터 계산 가능한 값을 Effect + setState로 만드는 **불필요한 Effect**가 없는가?
- [ ] fetch를 Effect에서 직접 다룬다면 race condition 처리(`ignore` 플래그 / `AbortController`)가 있는가?
      → 단, **서버 상태는 TanStack Query 로 옮기는 게 우선**이다. 손수 fetch+`ignore` 는 전환 후보로 표시.
- [ ] 의존성 배열이 객체가 아니라 **원시 필드**로 풀려 있는가? (`[params]` → `[params.a, params.b]`)
      객체를 그대로 두면 매 렌더 새 참조 → 무한 요청 위험.
- [ ] 검색어·범위처럼 입력 기반 네트워크 요청에 디바운스(또는 `useDeferredValue`)가 있는가?
      키스트로크마다 요청을 보내지 않는가?

### 4. Custom Hook 설계 — 한 문장으로 설명되는가
- [ ] 각 Hook이 **"무엇을 하는 Hook인지" 한 문장**으로 설명되는가?
      (예: `useOrders` = 주문 목록 서버 상태를 조회한다. 여기에 필터 UI 상태·라우팅까지 끼면 위험)
- [ ] 한 Hook이 너무 많은 책임을 지고 있지 않은가? — "500줄 컴포넌트"가 "500줄 Hook"으로
      옮겨간 것뿐이면 분리 대상. 페이지는 역할별 Hook을 **조합**만 한다.
- [ ] 이름에 역할이 드러나는가? (`useOrderStatusFilter` ✅ / `useData` · `useFetch` · `useHelper` ❌)
- [ ] Hook을 쓰지 않는 순수 함수에 `use` prefix를 붙이지 않았는가?
      (`formatOrderStatus`는 일반 함수 — `useFormatOrderStatus` ❌)
      참고: React 19의 `use(...)`는 빌트인 API(Promise/Context). 우리가 만드는 건 그 위의 `useXxx`.
- [ ] 이 Hook을 여러 컴포넌트에서 호출하면서 **상태가 공유될 것**이라 기대하고 있지 않은가?
      Custom Hook은 **로직 재사용**이지 **상태 공유**가 아니다 — 호출마다 `useState`가 다시 실행돼
      별개 인스턴스가 생긴다. 공유가 필요하면 props로 내리거나 외부 store(Context · Zustand ·
      Jotai · TanStack Query)로 옮긴다.

### 5. SRP — 한 가지 이유로만 변경되는가
- [ ] 컴포넌트 / Hook / 함수가 **하나의 변경 이유**만 갖는가?
      (컴포넌트=렌더링·상호작용 / Hook=도메인 로직·상태 조합 / Service=API 스펙 / Utils=계산 규칙)
- [ ] 도메인 규칙(상태 라벨 변환, 취소 가능 여부, 경과일 계산)이 컴포넌트 JSX 안에서 크게 계산되고
      있지 않은가? → 순수 함수(utils)로 분리 후보.
- [ ] 분리 기준은 "로직이 존재하는가?"가 아니라 **"이 로직이 별도의 변경 이유를 가지는가?"**.
      `const isSelected = selectedId === order.id` 같은 렌더링에 가까운 작은 계산은 컴포넌트 안에 둬도 된다.
- [ ] 입력 파싱·포맷 같은 순수 함수가 컴포넌트 안에 박혀 있지 않은가? → 단일 사용이어도 `utils/` 로.
- [ ] 타입(`type` / `interface`)이 `utils` 등 비-타입 모듈 안에 정의돼 있지 않은가? → `types/` 로.

### 6. DIP — 추상에 의존하는가
- [ ] Hook이 `axios` / `fetch` 구현에 **직접** 묶여 있지 않고 service(queryFn · Repository) 추상에
      의존하는가? (서버가 fetch→BFF→GraphQL로 바뀌어도 수정 지점이 service 한 곳으로 좁혀지는가)
- [ ] 컴포넌트가 서버 응답의 **세부 구조**(응답 래퍼, 중첩 필드)를 너무 많이 알고 있지 않은가?
      응답 형태 변환은 service에서 끝내고 컴포넌트엔 필요한 형태만 전달 → 서버 스펙 변경이 UI까지
      전파되는 것을 차단.

### 7. 컴포넌트 경계 — 어디서 자르나
- [ ] **변화의 경계(응집)**: 같이 바뀔 코드는 가까이, 따로 바뀔 코드는 다른 컴포넌트로 —
      "함께 변하는 것은 합리적인 범위에서 가까이". (변경 이유가 하나인지 자체는 §5 SRP)
- [ ] **구현 vs 조합 혼재**: 자식들을 **조합**하는 컴포넌트가 그중 하나의 구현(마크업·로직)을
      통째로 인라인하고 있지 않은가? 한 컴포넌트는 직접 구현이거나 조합이지, 한 레벨에서 둘을 섞지 않는다.
      (예: `<List><Header/> …카드 내부 article·title·desc 인라인… <Footer/></List>` → 카드도 컴포넌트로)
- [ ] **God Component 신호**: 아래가 보이면 분리 대상. 분리 *행동* 기준은 아래 *분리 후보 제안 기준* 참조.
  - 서로 무관한 상태가 한 곳에 (예: 목록 정렬 상태 + 모달 열림 상태)
  - data prop이 비정상적으로 많음 (7개 이상)
  - 로직·훅이 한 컴포넌트에 계속 쌓임
- [ ] **성급한 추상화**: YAGNI / Rule of Three — 쓰임이 2곳 이하인데 "나중을 위해" 미리 일반화하지
      않았는가? (오버 추상화 비권장 항목은 아래 *추출 비권장* 과 정합)

### 8. props 계약 — 적게, 역할대로
- [ ] **prop 개수**: 4~5개를 넘으면 리팩터 검토. (data prop 과다는 God Component 신호 — §7 과 연동)
- [ ] **이름 규칙**: 외부 주입 콜백은 `on~`, 내부 처리 함수는 `handle~`, boolean은 **긍정형**으로.
      (`disabled` ✅ / `notEnabled` ❌ — 이중 부정 금지, `code-quality.md` boolean 네이밍과 정합)
- [ ] **boolean 폭발 → union**: 상호배타 변형을 boolean 여러 개로 받지 않는가? **잘못된 조합은
      런타임이 아니라 타입에서** 막는다.
  - `Button({ primary, secondary, danger })` → `primary`·`secondary` 동시 지정이 타입상 가능 ❌
  - `variant: 'primary' | 'secondary' | 'danger'` ✅. 변형마다 필요한 props가 갈리면 discriminated union:
    `{ variant: 'text'; label: string } | { variant: 'icon'; icon: ReactNode }`
- [ ] **key 리셋**: 대상의 identity가 바뀔 때 내부 state를 초기화해야 하면, effect 동기화 대신
      **`key` 로 리마운트**했는가? (파생값을 state로 복사하지 말 것은 §2)
      (예: 대상이 바뀌어도 폼 입력이 남는다면 `<ReviewForm key={product.id} product={product} />`)
- [ ] **확장은 위임으로**: prop을 무한정 늘리는 대신 (a) `className` 병합(`cn`),
      (b) `ComponentPropsWithoutRef<'button'>` 로 HTML 속성 통째 위임,
      (c) polymorphic(`as` / `asChild`) — 셋 중 하나로 확장 지점을 열었는가?

### 9. 합성 — Context 전에 children
- [ ] **Context 전에 composition**: props를 깊게 내리거나 Context를 도입하기 전에 **children 합성**으로
      풀 수 있는지 먼저 봤는가? "props를 몇 단계 내린다"는 사실만으로 Context를 쓰지 않는다.
- [ ] **children vs slot**: 순서·위치가 고정돼야 하는 구조(Modal `header`/`footer` 등)는 자유로운
      `children` 보다 **이름이 정해진 slot prop**(사실상 위치가 고정된 props)이 적절한가?
- [ ] **Drilling vs Context**: 명확한 기준 없이는 Context를 쓰지 않는다.
  - **그냥 props** — 1~2단계로 짧게 내려가고 / 중간 컴포넌트도 그 값을 쓰고 / 전달 경로가 한눈에 보일 때
  - **Context** — 3단계 이상인데 중간은 전달만 하거나 / 멀리 떨어진 여러 컴포넌트가 같은 값을 쓰거나 /
    테마·로그인 사용자·언어처럼 앱 전역에서 읽는 값일 때

## 분리 후보 제안 기준

각 후보마다 **"분리했을 때 이점"과 "분리하지 않아도 되는 이유"를 함께** 제시한다.
무조건 쪼개라고 하지 않는다.

**분리 판단의 1순위는 SRP·응집도다 — 재사용 여부가 아니다.**
재사용은 분리의 *충분조건*일 뿐 *필요조건*이 아니다. 단 한 곳에서만 쓰여도 다음 중 하나면 분리한다.
- 서로 독립적으로 변하는 책임(컨트롤·섹션)을 여럿 들고 있을 때 (예: 카테고리 + 가격 + 재고 필터, 검색 + 정렬 + 보기모드)
- prop 개수가 과다할 때 — 한 컴포넌트가 너무 많은 값·콜백을 받을 때
- "하나의 흐름으로 설명되는 섹션"이 한 컴포넌트에 여러 개 묶여 있을 때

**추출 권장**
- 같은 상태 + 로직 조합이 2곳 이상에서 반복될 때
- 컴포넌트의 JSX와 무관한 로직이 컴포넌트를 크게 차지할 때
- 테스트하고 싶은 로직이 컴포넌트에 묶여 있을 때
- API 호출 · 필터 · 페이지네이션 · 검색 · validation처럼 하나의 흐름으로 설명되는 동작이 있을 때

**추출 비권장 (오버 추상화)**
- `useState` 하나만 감싸는 의미 없는 Hook (`useLoading` 류)
- 한 컴포넌트에서만 쓰이고, 분리하면 추적 비용만 커지는 로직
- Hook을 쓰지 않는 단순 순수 함수
- JSX나 스타일을 직접 반환하는 코드 (관심사 오염)

## 리포트 출력 형식

리뷰 결과는 아래 4개 블록으로 낸다.

**1) 관심사 분류표** — 현재 코드가 각 관심사를 어디에 두고 있는지

| 관심사 | 현재 위치(라인) | 비고 |
| --- | --- | --- |
| UI(JSX/스타일) | | |
| 클라이언트 상태 | | |
| 서버 상태 | | |
| 파생값 / 로직 조합 | | state화 여부 표시 |
| 도메인 로직 | | |
| API 통신 | | service 분리 여부 |

**2) 위반 지점 + 심각도**

- 🔴 높음 — 관심사 혼재로 변경/디버깅이 어려움, 파생값 state화, DIP 위반(컴포넌트가 fetch 직접 호출),
  God Component(무관한 상태 동거·data prop 7개+), boolean 폭발(잘못된 조합을 타입이 못 막음)
- 🟡 중간 — 불필요한 Effect, 책임이 1.5개인 Hook, 역할 안 드러나는 이름, props 4~5 초과,
  이름 규칙 위반(`on~`/`handle~`/긍정형 boolean), Context 남용(composition 미검토), 구현 vs 조합 혼재
- 🟢 사소 — 작은 계산 위치, 네이밍 다듬기, 코드를 재진술하는 불필요한 주석,
  확장 위임 여지(prop 누적), slot vs children 선택 다듬기

각 항목: `[심각도] 파일:라인 — 무엇이 / 왜 문제 / 어느 레이어로 가야 하는지`

**3) 분리 후보 표**

| 후보 이름 | 추출 위치(라인) | 한 문장 책임 | 분리 이점 | 분리 안 해도 되는 이유 |
| --- | --- | --- | --- | --- |

**4) 디렉토리 구조 — 이 레포의 표준**

아래는 권장이 아니라 **이 레포의 표준 디렉토리·명명**이다. 리뷰·리팩터 산출물은 반드시 이 명명을 따른다.

```
components/  ← JSX, 이벤트, 스타일, 작은 파생값
hooks/       ← 상태 관리, 도메인 로직, 서버 상태 조합
services/    ← API 호출, 요청/응답 변환 (Repository 역할). `api/` 로 쓰지 않는다.
utils/       ← 순수 함수 (포맷, 도메인 규칙). 타입을 여기 정의하지 않는다.
types/       ← interface / type
```

- API 레이어 디렉토리는 `services/` 로 통일한다 (`api/productApi.ts` 처럼 `api/` 로 두지 않는다).
- 순수 함수는 `utils/`, 타입은 `types/`. 모듈이 하나뿐이어도 이 명명을 지킨다.
- **권장 구조와 다른 산출물(`api/`, `utils.ts` 에 타입 혼재 등)을 내면 안 된다** — 내가 제안한 구조와 실제 결과가 어긋나지 않게 한다.
